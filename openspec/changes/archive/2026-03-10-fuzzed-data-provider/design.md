## Context

Fuzz targets in vitiate currently receive raw `Buffer` input. To test APIs that accept structured types (numbers, strings, objects), users must manually slice bytes and interpret them — a tedious and error-prone process. The LLVM FuzzedDataProvider pattern is the de facto standard for converting fuzzer-generated byte sequences into typed values, adopted by libFuzzer (C++), libfuzzer-sys (Rust), Jazzer (Java), and jazzer.js (JavaScript).

The jazzer.js project includes a TypeScript FuzzedDataProvider implementation (~750 lines, Apache-2.0 licensed) in `@jazzer.js/core`. Our implementation will follow the same LLVM semantics but as a standalone package, since future integration packages (`@vitiate/zod`, etc.) need to depend on the FDP without pulling in the full `@vitiate/core` dependency tree.

## Goals / Non-Goals

**Goals:**

- Provide a `FuzzedDataProvider` class with the full LLVM FDP API surface: booleans, integers (number + bigint), floats/doubles, strings, byte arrays, and element picking — all with range-constrained variants
- Match LLVM's split-buffer consumption semantics (individual values from end in LE, arrays/strings from front in BE) so the fuzzer's mutation engine can effectively explore both structural parameters and data content
- Ship as a standalone pure-TypeScript package (`@vitiate/fuzzed-data-provider`) with zero runtime dependencies
- Ensure the API is familiar to users of other FDP implementations (C++, Rust, Java, jazzer.js)
- Support `Uint8Array` input (not just Node.js `Buffer`) so the package works in non-Node environments

**Non-Goals:**

- Re-exporting FDP from `@vitiate/core` — the FDP is a standalone utility with its own versioning; re-exporting would couple core to it unnecessarily and defeat the purpose of a separate package
- Structured type generation beyond primitives (e.g., generating objects from schemas — that belongs in `@vitiate/zod` and similar integration packages)
- Custom mutator integration or feedback-aware generation — this is a stateless byte consumer
- Matching the jazzer.js API exactly where it diverges from LLVM semantics or where we can improve ergonomics

## Decisions

### 1. Package name: `@vitiate/fuzzed-data-provider`

**Decision:** Use `@vitiate/fuzzed-data-provider` as the npm package name, with the directory named `vitiate-fuzzed-data-provider` to match existing monorepo conventions (`vitiate-core`, `vitiate-engine`, `vitiate-swc-plugin`).

**Alternatives considered:**
- `@vitiate/fdp`: Too cryptic for users unfamiliar with the abbreviation. The full name is self-documenting and matches how other ecosystems refer to the concept.

### 2. Accept `Uint8Array` rather than `Buffer`

**Decision:** The constructor accepts `Uint8Array` (the web-standard type). Node.js `Buffer` is a subclass of `Uint8Array`, so it works without any adapter. This avoids a Node.js-only dependency and makes the package usable in browser-based fuzz harnesses or Deno/Bun environments.

Internally, use `DataView` for multi-byte reads (endian-aware) instead of Node.js `Buffer.readUIntLE`/`readUIntBE` methods.

**Alternatives considered:**
- Accept `Buffer` only (jazzer.js approach): Unnecessarily couples to Node.js. Since `Buffer extends Uint8Array`, accepting the supertype is strictly more general with no cost.

### 3. Split-buffer consumption strategy (matching LLVM)

**Decision:** Follow the LLVM/jazzer.js split-buffer design:
- **Individual values** (booleans, integrals, floats, doubles, `pickValue`): consumed from the **end** of the buffer, little-endian. These act as "parameters" guiding structure.
- **Bulk data** (byte arrays, strings, boolean arrays, integral arrays, number arrays): consumed from the **front** of the buffer, big-endian. These act as "content data."
- A `remainingBytes` counter tracks the unconsumed region between the two pointers.

This is critical for fuzzer effectiveness: the mutation engine can independently explore structural decisions (end bytes) and content (front bytes) without one corrupting the other.

**Alternatives considered:**
- Sequential consumption from front only: Simpler, but every structural parameter change shifts all subsequent data, making the fuzzer's hill-climbing ineffective.

### 4. API adjustments from jazzer.js

**Decision:** Keep method names identical to jazzer.js (which follow the LLVM convention), but adjust return types and parameter shapes where we can improve ergonomics:

| jazzer.js | vitiate | Change |
|-----------|---------|--------|
| `consumeBytes(maxLength): number[]` | `consumeBytes(maxLength): Uint8Array` | Return `Uint8Array` instead of `number[]` |
| `consumeRemainingAsBytes(): number[]` | `consumeRemainingAsBytes(): Uint8Array` | Return `Uint8Array` instead of `number[]` |
| `consumeString(maxLength, encoding, printable)` | `consumeString(maxLength, options?)` | Group encoding + printable into options object |
| `consumeRemainingAsString(encoding, printable)` | `consumeRemainingAsString(options?)` | Same options grouping |
| `consumeStringArray(maxArr, maxStr, enc, print)` | `consumeStringArray(maxArrayLength, maxStringLength, options?)` | Same options grouping |

All other methods retain identical names and signatures to jazzer.js: `consumeBoolean`, `consumeBooleans`, `consumeIntegral`, `consumeIntegralInRange`, `consumeIntegrals`, `consumeBigIntegral`, `consumeBigIntegralInRange`, `consumeBigIntegrals`, `consumeNumber`, `consumeNumberInRange`, `consumeNumbers`, `consumeFloat`, `consumeFloatInRange`, `consumeDouble`, `consumeDoubleInRange`, `consumeProbabilityFloat`, `consumeProbabilityDouble`, `pickValue`, `pickValues`.

Key rationale for the changes:
- `consumeBytes` / `consumeRemainingAsBytes` return `Uint8Array` instead of `number[]` — more efficient and idiomatic for binary data. Returning `number[]` (as jazzer.js does) is wasteful since each byte becomes a heap-allocated number. `Uint8Array` integrates directly with APIs that accept typed arrays and can use zero-copy slices when possible.
- String methods use an options object `{ encoding?, printable? }` instead of positional booleans — clearer at call sites, avoids the `(10, undefined, true)` pattern

### 5. Build tooling: tsup, mirroring vitiate-core

**Decision:** Use `tsup` for building, ESM-only output with `.d.ts` generation, matching the vitiate-core configuration. Target `node18` for consistency. Single entry point (`src/index.ts`).

### 6. Validation approach

**Decision:** Validate inputs at API boundaries and throw descriptive errors:
- Non-integer length/count parameters → `TypeError` with message indicating the parameter must be an integer
- `min > max` in range methods → `RangeError`
- Empty array in `pickValue`/`pickValues` → `RangeError`
- `maxNumBytes` out of bounds in `consumeIntegral` (must be 1-6) → `RangeError`

Use standard `TypeError` and `RangeError` instead of jazzer.js's custom `FloatLengthError` class. Standard error types are more idiomatic and work better with TypeScript type narrowing.

### 7. Exhausted buffer behavior

**Decision:** Match LLVM/jazzer.js semantics — when the buffer is exhausted:
- Range methods return `min`
- `consumeNumber()` returns `0`
- `consumeBoolean()` returns `false` (via `consumeIntegral` returning 0)
- Array/string methods return shorter results (down to empty)

This is the established convention across all FDP implementations and ensures deterministic behavior without throwing.

## Risks / Trade-offs

- **BigInt performance:** `consumeProbabilityDouble` uses BigInt division to get full 64-bit precision. This is slower than a pure float path but matches LLVM semantics. For most fuzz targets the FDP consumption time is negligible compared to target execution.
  → Mitigation: Profile if it becomes a bottleneck; could add a fast-path `consumeProbabilityDouble` using two 32-bit reads combined with float arithmetic.

- **DataView overhead vs Buffer methods:** Using `DataView` for endian-aware reads adds a small overhead compared to Node.js `Buffer.readUIntLE`. In practice this is negligible for the byte volumes involved in fuzzing (typically <64KB inputs).
  → Mitigation: Acceptable trade-off for platform independence. Can add a Node.js fast-path later if profiling warrants it.

- **API surface size:** The full LLVM FDP API is large (~25 public methods). This is intentional — it's a well-known interface that users expect to find complete.
  → Mitigation: Good documentation and TypeScript autocompletion make the large surface manageable.
